Chapter 4. Preliminaries

The adversary in this thesis controls up to f < N/2 Byzantine clients. Such
an adversary is free to relabel its own data, hand in any syntactically valid
ciphertext it likes, coordinate with other malicious clients, and listen in on every
public broadcast. The empirical part instantiates this model with targeted and
untargeted label flipping, at malicious rates f /N ∈ {10%, 20%, 30%, 50%}.

5.1

System Architecture

FIP-FL is built around four logical roles: the N clients, an Aggregation Server,
a FIP Verifier, and a Trusted Key Authority (or TKA). In a single round, the
clients put their Paillier ciphertexts on the wire, the Verifier turns each ciphertext
into one alignment score, only the accepted ciphertexts make it to the aggregation
server, and the TKA touches at most scalar scores or aggregate updates — never
an individual plaintext gradient.
This way of cutting the system keeps each role’s responsibility narrow. The
clients train locally and encrypt; the Verifier scores and filters; the server averages
whatever survives the audit; and the TKA does only the decryptions that the
protocol explicitly asks for. The Verifier itself is allowed to look at the encrypted
updates and the plaintext reference direction R(t) , but it is never given an
individual plaintext gradient. The TKA, on the other side, decrypts the scalar
FIP scores and the aggregate updates, but it does not see the audit logic at all.

5.2

System Setup and Key Distribution

The setup phase is run exactly once, before round 1 starts. Algorithm 1 keeps
the three kinds of state apart on purpose. The cryptographic state is the TKA’s
job, the model state begins from a clean global model on the server, and the
geometric state — the unit-norm reference direction — is derived from a small
piece of public bootstrap data and stored at the Verifier. From round 2 onwards
this same reference direction is refreshed from the accepted aggregate updates
rather than from any new public data.

5.3.5

Phase V: Reference Update and Model Distribution

In Phase V the model is moved by the averaged accepted gradient, and the
reference direction is refreshed through an exponential moving average with
momentum β. The refresh is followed by a normalisation step, which is what
guarantees that the FIP scores in the next round are still directional rather than
dominated by drift in ∥R∥. There is one corner case worth calling out: if the
audit rejects every client in a round, the model and the reference are simply held
unchanged for the next round. Algorithm 5 combines the secure aggregation
step, the model update, and the reference refresh in one block.
